## Challenge Description

The flag got lost, can you retrieve it?

> Challenge Link -> (LostFlag)[https://flagyard.com/labs/training-labs/3/challenges/1775ebb0-00a0-4c93-8450-7f7d4b13d240]


Looking at the decompiled code, we can see that the flag is lost after the function call `loadingAnimation` 

![Alt](/images/LostFlag/1.webp)

In the main function, after the `printf`call, the code loops from memory **`140006000`** to `0x1400060dc` stored at `_Argv` and XOR every 8 bytes, then it calls the `loadingAnimation(1000)` function that will run the animation for 1000 iterations (with a 200 ms `Sleep` each). That means **the data containing the flag has been decoded by the XOR loop before the heavy wait**.

I will use GDB for debugging, and if you are running the executable on linux machine, start the program under Wine in the background and pass the running process id to GDB

```bash
wine64 <file> & sleep 0.3
```

Find the PID of the Wine process that has loaded `load.exe`

```bash
ps aux | grep "load"
```

Attach gdb to that PID

```bash
gdb -p <PID>
```

We will break after the XOR loop, but before the `CALL loadingAnimation` which is the address `0x140005b95`of the MOV instruction that writes `0x3e8`into `_Argv` then read the memory of the first loop till the end of the loop. 

![Alt](/images/LostFlag/2.webp)

`x/s` in GDB, it shows strings at a memory address, and you can see the first char from the flag “F”, next try to print up to 512 bytes as C-strings (will show any ASCII sequences), keep printing the strings till you see the end of the flag format or the address `0x1400060dc` which indicates the end of the loop. 

![Alt](/images/LostFlag/3.webp)

Reconstruct the full flag string `FlagY{...}`
